# LuzGlass System Logic

## מטרת המערכת

מערכת LuzGlass מנהלת מפעל זכוכית לפי שני סוגי עבודה:

1. עבודה לפי הזמנות מלאות
2. עבודה לפי פריטים מתוך הזמנות

המערכת חייבת לשמור על זרימה יציבה בין:
לקוח → סקיצה → שרטט → OptyWay → יום עבודה → **תחנת בדיקה** → חיסום → מוכן לאיסוף → נאסף

⚠️ חיסום חייב לעבור דרך תחנת בדיקה. אין קיצור ישיר מיום עבודה לחיסום.

Firebase הוא מקור האמת היחיד לנתונים.

---

# עקרונות חובה

1. אין לשנות מבנה נתונים ללא אישור מפורש.
2. אין לבצע refactor רחב ללא אישור.
3. אין לשנות כמה קבצים יחד ללא צורך ברור.
4. אין לשבור פונקציות קיימות שעובדות.
5. אין להמציא לוגיקה חדשה אם כבר קיימת לוגיקה במערכת.
6. כל תיקון צריך להיות נקודתי ולפתור רק את הבעיה שהתבקשה.

# מודל עבודה: Orders Mode / Items Mode

## Orders Mode — מצב הזמנות

בשלבים הבאים המערכת עובדת לפי הזמנות מלאות:

- אדמין / דשבורד
- תור סקיצות
- שרטט
- OptyWay
- פורטל לקוח

במצב זה:
- ההזמנה היא יחידת העבודה המרכזית.
- מותר להציג את כל `o.items`.
- שינוי סטטוס בדרך כלל מתבצע על כל ההזמנה.
- אין להתייחס לפריטים כיחידת עבודה עצמאית, אלא אם המשתמש ביקש במפורש.

---

## Items Mode — מצב פריטים

בשלבים הבאים המערכת עובדת לפי פריטים מתוך הזמנה:

- בנה יום עבודה
- יום עבודה פעיל
- תחנת בדיקה (לפריטי חיסום בלבד)
- חיסום

במצב זה:
- יחידת העבודה היא פריט בודד מתוך הזמנה.
- מקור האמת לפריטים שנבחרו הוא `itemsSel`.
- אסור להציג או לעבד את כל `o.items` כאילו כולם נבחרו.

מבנה הבחירה:

```js
workDay.itemsSel = {
  [orderId]: [itemIndexes]
}

workDay.itemsSel = {
  "ord_123": [0, 2, 4]
}
